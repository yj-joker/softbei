# 检修任务模块 · 前端实现计划书

> 状态:草案,待你调整。本文件只做规划,不含最终代码。
> 技术栈沿用现有:Vue3 + Vite + Element Plus + vue-router,请求走 `src/api/request.js`(`/api` 代理 + session cookie + 大整数转字符串),配色/字体遵循「矿石白 + 克制蓝」浅色主题。

---

## 1. 目标与范围

把后端已就绪、但前端尚无入口的「检修任务」打通成完整可用模块:
**创建任务 → (异步)AI 生成检修步骤 → 逐步执行 → (异步)AI 验证步骤 → 关闭 → 沉淀为标准规程/知识图谱**。

其中两处异步(生成步骤、步骤验证)复用现有的 **WebSocket 通知 + Toast + 进行中托盘(RunningTasksTray)** 机制,无需新增后端代码。

---

## 2. 后端接口清单(全部已存在,基址 `/weixiu/task`)

| 用途 | Method | 路径 | 入参 | 返回 | 异步 |
|---|---|---|---|---|---|
| 创建任务 | POST | `/weixiu/task` | `MaintenanceTaskDTO` | `MaintenanceTaskVO` | ✅ 触发 `task.generate` → WS `TASK_GENERATED`/`TASK_GENERATE_FAILED`(data.taskId) |
| 重新生成 | POST | `/weixiu/task/{taskId}/retry` | — | — | ✅ 同上 |
| 开始执行 | POST | `/weixiu/task/{taskId}/start` | — | — | — |
| 提交/执行步骤 | POST | `/weixiu/task/{taskId}/steps/{stepId}/execute` | 步骤执行体(图片/备注) | `TaskStepRecordVO` | ✅ 触发 `task.step.verify` → WS `STEP_VERIFIED`(data.taskId, stepId) |
| 强制完成步骤 | POST | `/weixiu/task/{taskId}/steps/{stepId}/force-complete` | — | `TaskStepRecordVO` | — |
| 任务详情(含步骤) | GET | `/weixiu/task/{taskId}` | — | `MaintenanceTaskVO` | — |
| 任务列表(分页) | GET | `/weixiu/task` | `MaintenanceTaskQuery` | `PageResult<MaintenanceTaskVO>` | — |
| 步骤列表 | GET | `/weixiu/task/{taskId}/steps` | — | `List<TaskStepRecordVO>` | — |
| 沉淀为标准规程 | POST | `/weixiu/task/{taskId}/promote` | — | `Long`(procedureId) | — |
| 沉淀到知识图谱 | POST | `/weixiu/task/{taskId}/promote-to-graph` | — | — | — |
| 跳过沉淀 | POST | `/weixiu/task/{taskId}/skip-promotion` | — | — | — |
| 步骤 AI 答疑(流式) | POST | `/weixiu/task/{taskId}/steps/{stepId}/chat` | 提问体 | **SSE** `text/event-stream`(Flux) | 流式 |

辅助接口(已有):设备检索 `GET /weixiu/device/search`(创建任务选设备用,已封装在 `api/graph.js`);图片上传 `uploadImage`(`api/user.js`)。

## 3. 数据模型与状态机

**MaintenanceTaskVO**:`id, taskNumber, deviceId, deviceName, faultDescription, urgencyLevel(1-?), reportImages[], procedureId, procedureName, maintenanceLevel, status, generateMode, stepCount, reporterId, createdAt, updatedAt, graphExtraction, promotedProcedure, promotedGraph, steps[]`

**TaskStepRecordVO**:`id, taskId, sortOrder, title, content, safetyNote, requirePhoto, requireNote, estimatedMinutes, status, images[], note, completedAt, isCheckpoint, checkpointItems[], checkpointConfirmed, sources, generateConfidence, aiPass, aiConfidence, aiReason`

**MaintenanceTaskDTO**(创建):`deviceId, deviceName, faultDescription, urgencyLevel, reportImages[], maintenanceLevel, aiAdapt`

**任务状态** `status`:`CREATED → GENERATING → GENERATED → EXECUTING → CLOSED`,异常分支 `GENERATE_FAILED`。
**沉淀状态** `promotedProcedure` / `promotedGraph`:`PENDING / PROMOTED / SKIPPED`。
**步骤状态** `status`:`PENDING → SUBMITTED →(AI_PASSED | AI_REJECTED)→ COMPLETED`,可 `SKIPPED`。

状态 → 颜色建议(沿用主题语义色):
- 任务:GENERATING 蓝(转圈)、GENERATED/EXECUTING 蓝、CLOSED 灰、GENERATE_FAILED 红。
- 步骤:PENDING 灰、SUBMITTED 蓝(验证中)、AI_PASSED 绿、AI_REJECTED 橙/红、COMPLETED 绿、SKIPPED 灰。

## 4. 端到端流程(用户视角)

```
[列表页] 新建任务(选设备 + 故障描述 + 紧急度 + 报修图片 + 检修等级)
   │  POST /weixiu/task  → status=GENERATING
   │  ▶ trackJob(kind:'task', refId:taskId, title:'生成检修步骤')  // 进行中托盘
   ▼
(异步)AI 生成步骤 → WS TASK_GENERATED → Toast + 托盘清除 → 刷新详情(status=GENERATED)
   │  失败 → TASK_GENERATE_FAILED → Toast 红 → 详情显示「重新生成」
   ▼
[详情页] 开始执行  POST /{id}/start  → status=EXECUTING
   ▼
逐个步骤:阅读 content/safetyNote →(按 requirePhoto/requireNote 收集)→ 提交
   │  POST /{id}/steps/{stepId}/execute  → step=SUBMITTED
   │  ▶ trackJob(kind:'step', refId:stepId, title:'步骤AI验证')
   ▼
(异步)AI 验证 → WS STEP_VERIFIED → Toast + 托盘清除
   │  AI_PASSED(绿,可进入下一步) / AI_REJECTED(橙,展示 aiReason,可重做或「强制完成」)
   │  强制完成:POST /{id}/steps/{stepId}/force-complete
   │  疑问:步骤 AI 答疑抽屉(SSE /steps/{stepId}/chat)
   ▼
全部步骤完成 → 关闭任务 → 沉淀弹窗:
      沉淀标准规程 /promote · 沉淀知识图谱 /promote-to-graph · 跳过 /skip-promotion
```

## 6. 各页面/组件职责

**UserTasks.vue(列表)**
- `GET /weixiu/task` 分页 + 按 `status`/`deviceName` 筛选。
- 卡片/表格行:任务号、设备、故障摘要、状态徽章、步骤数、时间;点击进详情。
- 右上「新建任务」→ TaskCreateDialog。
- GENERATING 的任务行显示转圈,完成由 WS 自动刷新(可在 onMounted 订阅 notifyStore 或收到通知后 re-fetch)。

**TaskCreateDialog.vue**
- 设备:`searchDevices` 远程搜索下拉(回填 deviceId+deviceName)。
- 字段:faultDescription(必填)、urgencyLevel、maintenanceLevel、reportImages(复用 uploadImage)、aiAdapt 开关。
- 提交 `POST /weixiu/task` → 拿到 taskId → `notifyStore.trackJob({key:'task:'+taskId, kind:'task', refId:taskId, title:'生成检修步骤:'+deviceName})` → 跳转详情或留在列表。

**UserTaskDetail.vue(详情 + 执行)**
- `GET /weixiu/task/{id}` 渲染任务头 + 步骤列表(TaskStepCard)。
- status=GENERATING:骨架屏 + 「正在生成步骤…」;监听到 TASK_GENERATED 后 re-fetch。
- status=GENERATE_FAILED:显示「重新生成」→ `/{id}/retry`(再次 trackJob)。
- status=GENERATED:显示「开始执行」→ `/{id}/start`。
- status=EXECUTING:按 sortOrder 执行步骤(向导/逐卡)。
- 全部完成:出 PromoteDialog。

**TaskStepCard.vue / StepExecutePanel.vue**
- 展示 title/content/safetyNote(安全提示醒目)、estimatedMinutes、isCheckpoint 的检查项。
- 执行:按 requirePhoto/requireNote 收集图片/备注;检查点需 checkpointConfirmed。
- 提交 `/{id}/steps/{stepId}/execute` → step=SUBMITTED → `trackJob(kind:'step', refId:stepId)`。
- 收到 STEP_VERIFIED 后展示 aiPass/aiConfidence/aiReason;AI_REJECTED 提供「重做」或「强制完成」。

**StepChatDrawer.vue**
- 针对某步骤的 AI 答疑,`POST /steps/{stepId}/chat` 返回 SSE;复用 `AIChat.vue` 里 `getReader()+TextDecoder` 解析 `data:` 的写法 + AbortController 停止。

**PromoteDialog.vue**
- 三选项:沉淀标准规程 / 沉淀知识图谱 / 跳过;调用对应接口并刷新 `promotedProcedure`/`promotedGraph`。

## 7. 异步提示接入点(复用现有机制,零后端改动)

```js
// 1) 生成检修步骤(创建/重试后)
notifyStore.trackJob({ key:'task:'+taskId, kind:'task', refId:taskId, title:'生成检修步骤' })
//    完成:WS TASK_GENERATED(data.taskId) → notifyStore 「id 命中」自动清托盘 + 弹 Toast

// 2) 步骤 AI 验证(提交步骤后)
notifyStore.trackJob({ key:'step:'+stepId, kind:'step', refId:stepId, title:'步骤AI验证' })
//    完成:WS STEP_VERIFIED(data.stepId) → 自动清托盘 + 弹 Toast
```
> 注:Toast 是全局的,任何通知到达都会弹;托盘(转圈)才需要上面的 `trackJob`。
> 详情/列表页可在收到对应通知后 re-fetch 以刷新状态(可在组件内订阅 notifyStore,或简单地在 WS 回调里触发刷新)。

## 9. UX 要点

- 状态徽章统一:用 `constants/taskStatus.js` 把状态码映射成中文文案 + 主题色,避免散落硬编码。
- 安全提示(safetyNote)在步骤卡里用警示色块强调(高压/高温/旋转件等)。
- AI 验证中:步骤卡显示行内转圈 +「AI 验证中…」,与右下角托盘呼应。
- AI_REJECTED:展示 `aiReason` + 置信度,给「重做 / 强制完成」两条出路,避免卡死。
- 生成失败可重试,不让任务变成死状态。
- 全程浅色主题、圆角 8/12、柔和阴影,与现有页面一致。

## 10. 分阶段实施建议
